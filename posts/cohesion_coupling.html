Cohesion and Coupling

Posted by Helio Rui Ramos Freitas on Janurary 4, 2019

Cohesion is a way to say how much the functions in one module are related to each other. If the functions in a module have a strong connection, then they have a high cohesion because they work on the same problem. For example, the Model View Controller or MVC describes how to organize code separating it by purpose. Some modules will work handling the data, some will work handling application functionalities, and others will work on making the application look nice for the user. Adding code which handles the data in the module that works with the functionalities is a example of low cohesion because it will be inconsistent with the other code present in that module. Also, it will add more responsability to that module, and each module should have one responsability only. The cohesion is a measurement in a single module. It is a measurement of the functions are related to each other.

Coupling is how much the modules are related to each other, it evaluates how much they rely on each other to work. In other words, it is how strong their connection is. A good approach is to have a low coupling because frequently it means the code is well structured. I also think it may be considered good to have a low coupling because if there is a bug in a module, this bug has a lower chance of impacting on other modules. On the other hand, if they are very dependable of each other, the whole system may crash.

Low coupling is usually related with high cohesion and vice versa.

These two software quality metrics were invented by Larry Constantine back in the 1960s. Larry Constantine, is a software engineer and a college professor. He also made several contributions to the concepts and techniques used nowadays in software engineering.
